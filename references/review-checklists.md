## 1. PR Review 清单

适用场景：日常 Pull Request 审查，5-15 分钟快速评估。

### 1.1 概念完整性检查

- [ ] 新增 API/方法的命名风格与现有代码一致（动词时态、缩写习惯、语言）
- [ ] 新引入的抽象层次与周边模块对齐（不高不低）
- [ ] 错误处理方式统一（不混用 exception / error code / optional）
- [ ] 日志格式、级别使用符合项目惯例
- [ ] 注释语言一致（不出现中英文混杂）

### 1.2 Brooks 定律 — 耦合度检查

- [ ] 新增依赖是否必要？是否可用已有依赖替代？
- [ ] 模块间通信是否通过明确接口（非直接访问内部状态）
- [ ] 是否引入了循环依赖
- [ ] 共享可变状态的范围是否最小化
- [ ] 并发访问的临界区是否明确标注

### 1.3 第二系统效应 — 过度设计检查

- [ ] 是否为「可能的未来需求」添加了当前不需要的通用化
- [ ] 配置项数量是否合理（避免「配置爆炸」）
- [ ] 抽象层数是否超过 3 层（超过需要充分理由）
- [ ] 是否存在只被调用一次的接口/工具类
- [ ] 设计模式的使用是否必要，而非炫技

### 1.4 本质复杂度检查

- [ ] 新增的复杂度是「本质的」（问题域本身决定）还是「偶然的」（实现方式带来）
- [ ] 是否存在可以简化的算法或数据结构
- [ ] 嵌套深度是否超过 4 层
- [ ] 单个函数行数是否超过 50 行

### 1.5 可维护性检查

- [ ] 是否有充分的测试覆盖新增路径
- [ ] 关键决策是否有注释说明「为什么」
- [ ] TODO/FIXME 是否附带 issue 链接或时间戳
- [ ] 是否存在魔法数字/字符串（未命名常量）
- [ ] 变量/函数名能否自解释（不需要注释也能理解）

## 2. 新项目架构审查

适用场景：项目立项、架构设计评审，30-60 分钟深度评估。

### 2.1 概念完整性（架构层面）

- [ ] 核心设计理念是否有书面记录（ADR / 架构文档）
- [ ] 是否由单一核心架构师（或极少数人）主导设计决策
- [ ] 技术选型是否遵循一致的判断标准（而非多人各自决定）
- [ ] 模块划分原则是否清晰且一致应用

### 2.2 第二系统效应预防

- [ ] 团队中是否有成员「上一个项目」经验导致过度自信
- [ ] MVP 范围是否被严格限定
- [ ] 是否存在「我们这次要把 X 做得通用」的设计意图（风险信号）
- [ ] 架构是否针对已知需求设计，而非想象中的未来需求

### 2.3 外科手术团队结构

- [ ] 核心模块的「首席程序员」是否明确
- [ ] 接口设计是否由一人主导，避免「设计委员会」式妥协
- [ ] 团队规模是否在有效沟通范围内（<8 人）
- [ ] 各模块的所有者是否清晰（避免无主代码）

### 2.4 技术债务预算

- [ ] 是否预留了重构时间（建议 20% 的迭代容量）
- [ ] 「快速原型」代码是否被明确标记，并有计划替换
- [ ] 是否建立了技术债务追踪机制

### 2.5 焦油坑风险评估

- [ ] 是否存在「大泥球」风险（所有功能都往核心模块加）
- [ ] 数据模型是否足够稳定（频繁变更的数据模型是腐化的加速器）
- [ ] 外部依赖的版本策略是否明确

## 3. 遗留系统评估

适用场景：接手旧系统、评估维护成本，用于决策是否重写/重构/维持。

### 3.1 熵增程度评估

| 维度 | 低熵（健康） | 中熵（警戒） | 高熵（危险） |
|------|------------|------------|------------|
| 模块耦合 | 明确接口，低耦合 | 部分直接依赖 | 随意访问内部状态 |
| 命名一致性 | 统一规范 | 多种风格共存 | 无规律，随意命名 |
| 测试覆盖 | >70% | 30-70% | <30% |
| 文档 | 与代码同步 | 部分过时 | 严重滞后或缺失 |
| 构建时间 | <5 分钟 | 5-20 分钟 | >20 分钟 |

### 3.2 改动风险评估

- [ ] 核心路径是否有回归测试保护
- [ ] 是否存在「没人敢动」的模块（知识孤岛）
- [ ] 最后一次重大重构是多久前
- [ ] 原始作者是否仍在团队
- [ ] 是否存在未记录的「隐式约定」（如特定调用顺序）

### 3.3 重写 vs 重构决策矩阵

| 情况 | 建议策略 |
|------|---------|
| 核心逻辑正确，接口混乱 | 重构（绞杀者模式） |
| 技术栈严重过时，逻辑可迁移 | 渐进式重写 |
| 业务逻辑不清，文档缺失 | 先文档化再决策 |
| 性能瓶颈明确 | 针对性优化，非全量重写 |
| 所有人都想重写 | **警惕第二系统效应** |

## 4. 重构可行性评估

适用场景：规划重构方案前的风险评估。

### 4.1 安全重构前提

- [ ] 被重构模块有足够的测试覆盖（建议 >60%）
- [ ] 接口契约（输入/输出）有文档记录
- [ ] 有回滚方案（feature flag / 版本并行）
- [ ] 重构范围已明确界定（不要「顺手改」范围外的代码）

### 4.2 重构优先级评分

对每个候选重构项打分（各项 1-5 分，总分越高优先级越高）：

- **痛苦指数**：当前状态给开发带来多大阻碍？
- **改动频率**：这段代码多久被修改一次？（越频繁越值得重构）
- **影响范围**：重构后能改善多少相关代码？
- **风险系数**（反向）：重构失败的代价有多大？（越大分越低）
- **测试保护**：现有测试覆盖是否充分？

### 4.3 「做好扔掉的准备」原则应用

- [ ] 是否考虑过「直接重写这个模块」而非修补
- [ ] 原型/实验性代码是否已通过 `@prototype` / `TODO: replace` 标记
- [ ] 新版本与旧版本是否可以并行运行（减少迁移风险）

## 5. 依赖管理审查

适用场景：依赖升级、引入新依赖、安全审计。

### 5.1 引入新依赖评估

- [ ] 此依赖解决的问题能否用已有依赖解决？
- [ ] 维护状态：最近 6 个月是否有提交？是否有活跃维护者？
- [ ] 许可证与项目兼容？
- [ ] 包体积是否合理（前端项目尤其关注）
- [ ] 是否有已知的安全漏洞（CVE）？
- [ ] 传递性依赖数量是否可接受？

### 5.2 依赖版本策略

| 类型 | 推荐策略 |
|------|---------|
| 核心框架 | 锁定 minor 版本，定期升级 |
| 工具类库 | 允许 patch 自动升级 |
| 安全相关 | 立即升级，不允许延期 |
| 内部包 | 明确版本，避免 `latest` |

### 5.3 依赖健康度检查

- [ ] 是否存在版本冲突（同一包的多个版本）
- [ ] 是否存在已废弃（deprecated）的依赖
- [ ] 直接依赖与间接依赖是否分层管理
- [ ] lock 文件是否提交到版本控制

## 6. 常见反模式速查表

按 Brooks 原则分类的代码坏味道，快速识别用。

### 违反概念完整性

| 反模式 | 症状 | 修复方向 |
|-------|------|---------|
| 风格分裂 | 同一文件中多种命名风格 | 统一规范，自动格式化 |
| 抽象不一致 | 部分用 Repository 模式，部分直接 SQL | 统一数据访问层 |
| 错误处理混乱 | exception + error code + null 混用 | 选定一种策略全面推行 |
| 多语言注释 | 注释时中时英 | 团队统一语言 |

### 违反 Brooks 定律（高耦合）

| 反模式 | 症状 | 修复方向 |
|-------|------|---------|
| 意大利面条代码 | 函数调用图混乱，无明显层次 | 引入分层架构 |
| 神类 (God Class) | 单个类承担所有职责 | 单一职责原则拆分 |
| 特性嫉妒 | 方法大量使用其他类的数据 | 移动方法到正确位置 |
| 循环依赖 | A→B→C→A | 引入接口或中间层 |

### 第二系统效应（过度设计）

| 反模式 | 症状 | 修复方向 |
|-------|------|---------|
| 过早抽象 | 抽象层在整个项目中只有一个实现 | YAGNI，等需求明确后再抽象 |
| 配置地狱 | 行为由 50+ 配置项控制 | 约定优于配置 |
| 过度通用化 | 「框架」比使用它的业务代码更多 | 降低通用化程度 |
| 工厂的工厂 | 多层工厂/注册表/策略嵌套 | 简化对象创建 |

### 本质复杂度失控

| 反模式 | 症状 | 修复方向 |
|-------|------|---------|
| 深嵌套 | if/for 嵌套 >4 层 | 提前 return，提取函数 |
| 长方法 | 单个函数 >80 行 | 按职责拆分 |
| 参数过多 | 函数参数 >5 个 | 引入参数对象 |
| 魔法数字 | 代码中散落 42, 86400, 0.75 | 命名常量 |

### 焦油坑（渐进腐化）

| 反模式 | 症状 | 修复方向 |
|-------|------|---------|
| 大泥球 | 无法描述核心模块的职责边界 | 绞杀者模式逐步重构 |
| 破窗效应 | 一处乱，处处乱 | 设立质量门槛并强制执行 |
| 注释墓地 | 大量被注释掉的旧代码 | 删除，依赖 git 历史 |
| TODO 坟场 | 大量无日期、无 owner 的 TODO | 清理或转为 issue |
