# Brooks-Lint v0.3 Design Spec
# Large Organization Extensions

**Date:** 2026-03-27
**Status:** Approved
**Scope:** 面向多团队工程组织的能力扩展

---

## 背景

Brooks-Lint v0.2 覆盖了《人月神话》7 个核心维度，面向个人开发者和小团队。
v0.3 面向更大规模的工程组织（多团队、有 PM、有架构委员会），
补充书中尚未覆盖的三个重要领域：

1. **文档完整性**（Ch.15 "另一面" + Ch.7 巴别塔）
2. **Conway 法则检验**（Ch.7 巴别塔 — 组织结构镜像）
3. **ADR 健康度**（Ch.10 项目手册 — 架构决策记录）

---

## 设计决策

### 方案选择：零新文件，扩展四个现有文件

**决策：** 不新增 guide 文件，扩展现有文件。

**放弃的替代方案：**
- 新增独立 guide 文件（如 `documentation-guide.md`、`conway-guide.md`）
- 新增 Mode 4/5/6 和对应斜杠命令

**原因：**
- 现有架构模式已明确：`brooks-principles.md` 存放跨模式评分 rubric，
  三个 guide 文件存放各模式的检查清单
- 文档维度 rubric 与现有 7 个维度格式完全一致，天然属于 `brooks-principles.md`
- Conway 检验是架构审计的子集，ADR 是知识债务的子类，均有现成归宿
- 新建文件会打破 SKILL.md 的 on-demand 加载模式，形成孤儿文件
- 符合 Brooks 的概念完整性原则：宁可系统缺少某些功能，
  也要反映一种连贯的设计哲学

## 第 8 维度：Documentation Completeness

### 来源

- Ch.15 "另一面"："程序的另一面是它需要的文档……程序必须被使用，这需要文档。"
- Ch.7 巴别塔："巴别塔的失败不是缺乏愿景，而是缺乏沟通。"

### 结构

两个子维度，各自独立评分，报告中分拆显示：

```
| Documentation | ⬛⬛⬜⬜⬜ 2/5 | 代码级 2/5 · 架构级 1/5 |
```

整体分 = (代码级 + 架构级) / 2，权重与其他维度相同。

### Overall Health 计算更新

- 现有：分母 9（Conceptual Integrity 和 Communication Overhead 各权重 2，其余权重 1）
- v0.3：分母 10（Documentation 权重 1，加入计算）

### 8a. Code-Level Documentation 评分表

| 分数 | 含义 | 可观察证据 |
|------|------|-----------|
| 5 | 典范 | 每个公共接口有完整说明。复杂逻辑有 why 注释。新开发者无需追溯实现即可使用任意模块 |
| 4 | 良好 | 主要公共接口有说明，1-2 处非关键路径缺失 |
| 3 | 可接受 | 核心接口有说明，边缘功能缺失。复杂逻辑部分有解释 |
| 2 | 需关注 | 公共接口普遍缺乏说明。理解模块必须读实现 |
| 1 | 危急 | 几乎无文档。代码库对新人不可访问 |

**低分症状：**
- 公共函数无注释，调用者必须读实现才能理解行为
- 复杂算法或业务规则无任何解释
- 参数含义依赖命名猜测（`processData(x, y, flag)`）
- 注释描述 what（代码本身已说清楚），而非 why

### 8b. Architecture-Level Documentation 评分表

| 分数 | 含义 | 可观察证据 |
|------|------|-----------|
| 5 | 典范 | ADR 存在且与当前实现一致。跨团队接口有规范文档。新团队成员可通过文档理解系统，无需访谈原作者 |
| 4 | 良好 | 主要架构决策有记录，1-2 个重要决策缺失 ADR |
| 3 | 可接受 | 有部分设计文档但不完整或有滞后。关键接口有说明 |
| 2 | 需关注 | 架构文档几乎不存在。跨团队接口靠约定俗成 |
| 1 | 危急 | 无任何架构级文档。系统知识完全在人脑中 |

**低分症状：**
- 架构决策无记录（"为什么用这个框架？" 无人能答）
- 跨团队接口无规范文档，只靠口头协议
- 系统级 README 不存在或描述过时架构
- 重要决策只存在于某人的 Slack 消息里

## 各模式的 Dimension 8 检查清单

### Mode 1（PR Review）— `pr-review-guide.md`

**Quick skip（代码级）：** PR 新增的所有公共函数/接口都有说明？复杂逻辑有 why 注释？两者皆是 → ✅ 代码级跳过。

**Quick skip（架构级）：** PR 不涉及架构决策或跨团队接口变更？→ ✅ 架构级 N/A。

代码级检查：
- 新增的公共函数/方法是否有说明调用者行为的注释？
- 复杂逻辑或非显而易见的实现是否有 why 注释（而非 what）？
- 修改了已有公共接口，是否同步更新了对应文档？
- 新增参数名称是否自解释，或是否有说明？

架构级检查（仅当 PR 涉及重要架构决策时）：
- 是否需要新建或更新 ADR？
- 跨团队接口变更是否有对应的接口规范更新？

### Mode 2（Architecture Audit）— `architecture-guide.md`

代码级检查：
- 抽样 3-5 个公共模块接口：是否有足够文档让调用者无需读实现？
- 系统中是否存在"只有原作者能解释"的复杂逻辑？
- 注释是否在解释 why（设计决策）而非 what（代码本身已说明）？

架构级检查：
- 是否存在 ADR 或等价的设计决策记录？
- 关键架构决策（框架选型、模块划分、数据流设计）是否有文档？
- 跨团队接口是否有规范文档，还是依赖口头约定？
- 系统级 README 是否描述当前架构（而非历史版本）？

### Mode 3（Tech Debt）— `debt-guide.md`

代码级识别特征：
- 公共接口无文档，理解模块必须读实现
- 复杂业务规则内嵌于代码，无任何说明 why 的注释
- 注释与代码不一致（描述旧行为）

架构级识别特征：
- 架构决策无 ADR，只活在原作者记忆中
- 跨团队接口文档缺失或过时
- 系统 README 描述的架构已被重构，文档未更新

严重度：
- **高：** 核心业务逻辑无文档 + 无 ADR → 新人上手成本极高，修改风险无法评估
- **中：** 有部分文档但覆盖不均，架构级文档有明显滞后
- **低：** 仅工具/脚本类代码缺文档，核心路径覆盖良好

## Conway 法则检验

**落点：** `architecture-guide.md` Step 5（现有 Step 1-4 之后）

**触发条件：** 用户提供团队结构信息时执行；未提供则跳过并说明原因。

**为何需要显式团队信息（不推断）：**
Brooks 在 Ch.7 强调：巴别塔的失败是缺乏显式沟通结构。
从代码推断团队结构是隐式考古，而非 Conway 检验。
检验需要两边信息：代码结构 + 组织结构，缺一不可。

### 信息请求

若用户未提供团队结构，Claude 请求：
> "Conway 法则检验需要了解团队划分。请简要描述：哪些团队负责哪些模块？"

### 检验内容

**5a. 结构吻合度**
- 团队边界与模块边界是否吻合？
- 跨团队依赖方向是否合理（单向 vs 双向）？

**5b. 巴别塔风险识别**

没有公共语言的表现：
- 跨团队接口是否有统一的数据格式/协议规范？
- 不同团队的模块是否使用不同的命名约定、错误处理策略？

没有组织的表现：
- 是否存在"无主模块"（没有团队明确负责的共享代码）？
- 跨团队变更是否需要协调多个团队才能完成一个功能？

### 报告输出格式

```
## Conway's Law Check

**团队结构：** [用户提供的描述]
**结构吻合度：** ✅ 良好 / ⚠️ 部分问题 / ❌ 严重错位

| 检验项 | 状态 | 发现 |
|--------|------|------|
| 团队边界 ↔ 模块边界吻合 | ✅/⚠️/❌ | [一行描述] |
| 跨团队依赖方向 | ✅/⚠️/❌ | [一行描述] |
| 公共语言（接口规范） | ✅/⚠️/❌ | [一行描述] |
| 无主模块风险 | ✅/⚠️/❌ | [一行描述] |

**巴别塔风险等级：** 低 / 中 / 高
[2-3 句总结最关键的发现]
```

## ADR 健康度（知识债务子类）

**落点：** `debt-guide.md` Category 4 Knowledge Debt 下新增子类

### 识别特征

- 代码库中不存在任何 ADR 或等价文档（`docs/decisions/`、`docs/adr/`、`docs/rfcs/` 均不存在）
- ADR 存在但与当前实现不一致（记录的决策已被推翻，文档未更新）
- 重要的架构转折点无对应 ADR
- ADR 只记录"做什么"，未记录"为什么"和"放弃了哪些替代方案"

### 检测方法

1. 查找 `docs/`、`RFC`、`decisions/`、`adr/` 等目录是否存在
2. 若存在，抽查 2-3 个 ADR：内容是否与当前代码一致？
3. 识别代码中的重大架构特征，检验是否有对应 ADR

### 严重度

- **高：** 系统有多次明显的架构演化痕迹，但零 ADR 记录
- **中：** 有部分 ADR 但覆盖不全，或 ADR 存在明显滞后
- **低：** ADR 体系基本完整，仅个别次要决策缺失记录

### 偿还方式

- 不要补写历史 ADR（成本高、准确性低）
- 从当下开始：下一个重要决策必须配套 ADR
- 对现有"无人敢动"的模块，写"反向 ADR"——记录当前理解，哪怕不完整
- ADR 最小可行模板：
  - **背景：** 当时面对什么问题
  - **决策：** 我们选择了什么
  - **放弃了什么：** 考虑过但未选的替代方案
  - **后果：** 预期的权衡与影响

### 报告格式（Knowledge Debt 行细化）

| Category | Severity | Key Evidence | Estimated Impact |
|----------|----------|-------------|-----------------|
| Knowledge Debt — ADR | High | 零 ADR，3 次明显架构演化无记录 | 新团队上手需 2-3 周访谈 |
| Knowledge Debt — 人员 | Medium | 支付模块仅一人理解 | Bus factor = 1 |
