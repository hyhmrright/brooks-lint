## 2. Eval Benchmark Suite

### 2.1 Design Principles

- **Primary axis: Decay Risk coverage.** Every scenario targets a specific decay risk. Mode and language are secondary dimensions that rotate naturally.
- **Each risk gets positive, borderline, and clean scenarios** to verify both detection accuracy and false-positive resistance.
- **Language rotation**: Python, TypeScript, Go, Java cycle through scenarios — no language-dedicated evals needed.
- **Mode is derived from scenario form**, not forced. Single-file diffs → PR Review; multi-module structures → Architecture Audit; narrative descriptions → Tech Debt; test files → Test Quality Review.

### 2.5 File Format

Preserve existing `evals/evals.json` format. Each scenario:

```json
{
  "id": <number>,
  "name": "<risk-id>-<severity>-<lang>",
  "prompt": "<code or description>",
  "expected_output": "<what the report must contain>",
  "files": []
}
```

Naming convention: `r1-critical-python`, `t4-positive-typescript`, `r5-clean-go`, etc.

### 2.6 Expected Output Criteria

Each `expected_output` must specify:

1. **Which decay risk** should be identified (or explicitly "no findings" for clean scenarios)
2. **Severity level** expected (Critical / Warning / Suggestion)
3. **Iron Law completeness**: Symptom, Source (book + concept), Consequence, Remedy — all four present
4. **For clean scenarios**: no false positives for the target risk

### 2.7 Backward Compatibility

Existing 4 evals (id 1-4) are kept as-is. New scenarios start from id 5. The original scenarios serve as integration-level smoke tests; new scenarios are focused unit-level probes of individual risks.

### 3.3 Content Source

Gallery content is generated by running brooks-lint on eval scenario inputs. The workflow:

1. Pick the most visually compelling eval scenarios (2-3 per Mode)
2. Run brooks-lint on each
3. Copy the output verbatim into gallery.md
4. Add minimal context headers (language, target risk, severity)

### 3.4 Gallery Principles

- **No manual editing of output** — what the skill produces is what goes in the gallery
- **Include at least one "clean" report** — shows users what healthy code looks like through the lens
- **Include the Mermaid diagram** — this is a visual differentiator, must be in the gallery
- **Keep it under 500 lines** — gallery should be browsable, not exhaustive

## 5. Build Sequence

1. **Design eval scenarios** — write all 33 scenario prompts + expected output criteria in `evals/evals.json`
2. **Run evals** — execute each scenario through brooks-lint, collect output
3. **Curate gallery** — select ~10 best outputs, assemble `docs/gallery.md`
4. **Update README** — add gallery link
5. **Update CHANGELOG** — document the eval expansion and gallery addition

## 6. Success Criteria

- `evals/evals.json` contains 33+ scenarios covering all 12 decay risks
- Every decay risk has at least one Critical and one Clean scenario
- `docs/gallery.md` exists with 8-10 real outputs organized by Mode
- Gallery includes at least one Mermaid diagram rendering
- README links to gallery
- All four languages (Python, TypeScript, Go, Java) appear in both evals and gallery
